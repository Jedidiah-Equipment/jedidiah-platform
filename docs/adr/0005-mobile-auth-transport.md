# Platform-Appropriate Auth Transports

The mobile app (`pkg/mobile`, Expo/React Native) and the web app share one better-auth backend — one session model, one set of endpoints, one sign-in eligibility check. Both use better-auth's **cookie-based** session, but the cookie is held in a platform-appropriate place. Web relies on the browser's httpOnly cookie jar. Mobile uses better-auth's Expo integration (`@better-auth/expo` client plugin + the server-side `expo()` plugin): there is no browser cookie jar on a device, so the integration persists the session cookie in `expo-secure-store` (Keychain/Keystore) and replays it as a `Cookie` header (via `authClient.getCookie()`) on each request.

We deliberately did not unify both clients on a **bearer-token** transport (better-auth's `bearer` plugin) for the sake of a single pattern. In a browser, an httpOnly cookie keeps the session token out of reach of JavaScript, so an XSS bug cannot exfiltrate it; a bearer token would have to live in browser-reachable storage, where XSS can steal it. Switching web to bearer would therefore downgrade the more-exposed surface for no real gain — mobile already has a working cookie transport through the Expo integration's SecureStore-backed cookie, so there is nothing to "unify" toward. This is one backend with one cookie session, stored where each platform can protect it, not two auth patterns.

Adding the mobile app required registering its deep-link scheme in `AUTH_TRUSTED_ORIGINS` and adding the `expo()` plugin to the better-auth `plugins` array in `pkg/api/src/auth/auth.ts`. Sign-in eligibility is unchanged: phase-one mobile serves the same already-eligible roles as web (`bay-operator` remains not sign-in eligible).
