# Schedule windowing

The Bay schedule is read through one shared query, `jobs.listBays`, that every scheduling surface depends on — the web Gantt board and dashboard widgets, and the mobile Job List, Bay List, and Job Detail. Today it loads every Bay's entire Slot queue, projects all of it in memory, and resolves a product/customer summary for every Job that ever touched the board. Because Slots are never pruned (dates are relative to a fixed `scheduleOrigin`, so a completed Slot cannot be removed without shifting its successors), that read grows without bound as lifetime throughput accumulates. The cost that bites first is not server memory but the payload shipped to every client, the browser render of it, and the all-history summary join — all paid on every board open.

We keep the projection model of ADR 0003 exactly: Slot dates stay derived, never stored, and the server remains the authority that projects the full queue. What changes is what leaves the server. The client no longer receives the whole queue; it receives a **window** of already-projected Slots to render, and asks the server whenever it needs a projection-dependent preview.

The window is asymmetric because the problem is asymmetric. Forward work is self-limiting — a shop can only book so far ahead, so the number of future Slots is bounded by bookings, not by calendar span, and every consumer needs all of it. Backward work is the only unbounded axis: completed Jobs pile up over years. So the default read is **job-centric**, not a symmetric calendar window. It returns all Slots of any Job with an unfinished Work Slot (max projected `endDate` past plant "today"), unioned with any Slot on or after a `from` floor that defaults to today. A Job that is fully off the floor is dropped. This bounds the shared read to work-in-progress plus forward bookings regardless of how much history exists, and stays correct for Job progress and route timelines, which need a long-running Job's already-done Slots even when those fall weeks in the past — a fixed calendar back-buffer would corrupt them.

Completeness reuses the existing rule (ADR 0003 / Job Days-Left): a Job is off the floor iff every one of its Slots has projected `endDate` on or before plant "today". There is no actuals tracking, so a behind-schedule Job past its planned window counts as complete, exactly as the board already treats it. Because a Job spans Bays, the server computes the Job-completion map from the full projection before trimming any Bay's Slots; a Bay cannot decide its returned Slots in isolation. Only the `slots` array is windowed — each Bay's `nextAvailableDate` and other queue-end scalars are always computed from the full projection, so a Bay whose work is all complete returns an empty in-view `slots` array with an accurate `nextAvailableDate` rather than looking mis-scheduled.

The Gantt reaches into history through the same query with its own `from`. `listBays({ from })` returns active Jobs whole unioned with Slots ending on or after `from`, so a single call already carries both the forward active work and the historical Slots back to `from`. Non-Gantt consumers call the bare `listBays()` (`from` defaults to today) and land on their own cache entry, untouched as the Gantt scrolls. The Gantt's `from` snaps to month buckets and only ever extends backward — scrolling forward within loaded range is free — and the server clamps `from` to roughly twelve months before today so a client cannot re-arm the unbounded read on demand.

Projection-dependent previews move server-side, because the client no longer holds the queue to project against. The Working Calendar (Off-Days and Bay Calendar Exceptions) and `nextAvailableDate` are bounded facts and stay on the client, so the date picker keeps disabling off-days and computing bounds locally with no round trip. Slot moves stop being previewed at all: a move has nothing to decide, so it becomes a direct mutation followed by a windowed refetch. The only surviving preview is the insert-seeds family — the booking dialog's placement text and the multi-Bay ghost bars — served by one endpoint, `jobs.previewSchedule({ from, seeds })`, that applies the pending seeds through the same placement seam the booking mutation uses and returns the full windowed, reprojected affected Bays plus ghosts and placements. It cannot return a thin delta, because inserting a seed reprojects every downstream Slot's date, which the windowless client cannot recompute. Client previews remain advisory; server resolution stays authoritative.

## Considered Options

- **Store the projection (dates on Slots, or a materialized view)** — rejected as the source of truth. It simplifies reads but the projection engine survives anyway for previews, and it adds write amplification (one Off-Day toggle re-dates every downstream Slot across every Bay) plus a drift surface between stored dates and the pure projection. Acceptable only as a rebuildable cache, never canonical.
- **Absolute dates as the truth (drop sequence + duration)** — rejected. It relocates the projection from read-time to write-time rather than removing it, keeps the full engine on the client for previews, and makes an org-wide calendar change a fan-out rewrite across every Bay. Net complexity rises; the pure "sequence + duration is the whole truth" model is worth protecting.
- **Symmetric client-supplied date window for all consumers** — rejected. It couples the mobile Job List to the Gantt's viewport and, with a fixed calendar back-buffer, drops the already-done Slots that a long-running Job's progress and route timeline still need.
- **A thin preview delta the client merges** — rejected. A mid-queue insert reprojects downstream Slot dates, which the windowless client cannot compute, so the preview must return the reprojected windowed Bays.
- **History compaction by advancing `scheduleOrigin`** — deferred, not rejected. Advancing a Bay's origin past a prefix of Slots whose Jobs are fully complete, and renumbering the survivors, reproduces identical dates for them and would bound the stored working set, not just the read. It composes with windowing and can be added later; a read-side archive flag would preserve completed-Job route history if it is ever pruned. Out of scope here because windowing alone defuses the read cost, and compaction adds a background sweep, a cross-Bay completeness pass, and an origin-as-moving-watermark semantics change.
- **Compute projection in SQL to bound server memory too** — deferred. Keeping the full projection server-side leaves the server load at O(all history), which is cheap for a long time. Windowing the *contract* now creates the seam to later swap the implementation to a SQL prefix-sum over `durationDays` that hydrates only the window, with no change visible to any client.
