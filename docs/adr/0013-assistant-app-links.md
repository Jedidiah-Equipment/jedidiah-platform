# Assistant App Links Are Canonical Web Hrefs Resolved Per Client

The Assistant's tools attach internal links (`links.app`) to the entities they return, and the model writes those hrefs directly into streamed message text as Markdown links (`[Craig Stokes](/customers/456/edit)`). With the Assistant now on mobile as well as web, the link language had to be settled: the hrefs are baked into message text, so whatever the server emits is what every client receives.

We decided the **web-shaped hrefs stay canonical** — `/quotes/:id/edit`, `/products/:id/edit`, `/customers/:id/edit`, `/jobs/:id`, plus the two `/api/...` download paths — and **each client resolves them at interaction time**. Web navigates them via its router as today. Mobile parses a tapped href and switches on the parsed kind: quotes, products, and jobs push the matching native screen, the two download hrefs open the in-app document viewer, and a kind with no mobile surface (currently customers) renders as plain text with no link affordance, as does any href the installed binary does not recognize. The href factories and the parser (`parseInternalAppHref`, returning a discriminated union) live together in one module in `@pkg/schema` (re-exported from `@pkg/ai`), so producing and parsing cannot drift and a TypeScript exhaustiveness check forces every link kind to be explicitly mapped or unsupported per client.

Rejected alternatives: **platform-neutral hrefs** (e.g. dropping `/edit`) would churn web and the prompt suite purely for symmetry; **server-side platform forking** (client declares its platform, server emits native links) would make the same conversation render different text per device and would bake device-specific links into any future persisted chat history. The accepted cost is that mobile depends on web's URL shapes — mitigated by the shared factory/parser module and its exhaustiveness check rather than runtime hope.

OS-level deep linking (universal links / Android App Links opening the native app from external URLs) is explicitly not part of this decision; App Links here only originate and resolve inside a signed-in client.
